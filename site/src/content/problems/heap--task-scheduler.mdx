## Problem Statement

You are given an array of CPU `tasks`, each labeled with a letter from A to Z, and a number `n`. Each CPU interval can be idle or allow the completion of one task. Tasks can be completed in any order, but there's a constraint: there has to be a gap of **at least** `n` intervals between two tasks with the same label.

Return the **minimum** number of CPU intervals required to complete all tasks.

 

**Example 1:**

**Input:** tasks = ["A","A","A","B","B","B"], n = 2

**Output:** 8

**Explanation:** A possible sequence is: A -> B -> idle -> A -> B -> idle -> A -> B.

After completing task A, you must wait two intervals before doing A again. The same applies to task B. In the 3^rd^ interval, neither A nor B can be done, so you idle. By the 4^th^ interval, you can do A again as 2 intervals have passed.

**Example 2:**

**Input:** tasks = ["A","C","A","B","D","B"], n = 1

**Output:** 6

**Explanation:** A possible sequence is: A -> B -> C -> D -> A -> B.

With a cooling interval of 1, you can repeat a task after just one other task.

**Example 3:**

**Input:** tasks = ["A","A","A", "B","B","B"], n = 3

**Output:** 10

**Explanation:** A possible sequence is: A -> B -> idle -> idle -> A -> B -> idle -> idle -> A -> B.

There are only two types of tasks, A and B, which need to be separated by 3 intervals. This leads to idling twice between repetitions of these tasks.

 

**Constraints:**

- `1 <= tasks.length <= 10^4`
- `tasks[i]` is an uppercase English letter.
- `0 <= n <= 100`

## Solution

```python
from typing import List


class Solution:
    def leastInterval(self, tasks: List[str], n: int) -> int:
        if n == 0:
            return len(tasks)
        task_count = [0] * 26
        for task in tasks:
            task_count[ord(task) - ord('A')] += 1
        task_count.sort()
        max_task = task_count[-1] - 1
        idle_slots = max_task * n
        for i in range(24, -1, -1):
            idle_slots -= min(max_task, task_count[i])
        return len(tasks) + max(0, idle_slots)
```


## Explanation

### Approach: Greedy Algorithm

This solution uses a **Greedy** strategy to make locally optimal decisions that lead to the global optimum.

### Intuition

The task with the highest frequency determines the skeleton of the schedule. If the most frequent task appears `max_count` times, then those occurrences create `max_count - 1` gaps that each need at most `n` cooldown slots. Other tasks can be used to fill those gaps, and any leftover gaps become idle time.

### Algorithm

Here's the step-by-step approach:

1. Count the frequency of every task.
2. Sort the counts so the most frequent task is last.
3. Compute the initial number of idle slots created by the most frequent task: `(max_count - 1) * n`.
4. Use the remaining task frequencies to fill those idle slots, then return `len(tasks) + max(0, idle_slots)`.

### Code Breakdown

`max_task` represents the number of full gaps between repeated occurrences of the most frequent task. For every other task count, the code subtracts how many of those gaps it can fill, up to `max_task`, because each task can contribute at most one occurrence per gap.

### Why This Works

- **Dominant-task constraint:** No schedule can avoid the spacing forced by the most frequent task.
- **Gap-filling logic:** Any other task placed into those gaps reduces necessary idle time without violating cooldowns.
- **Exact final length:** If all gaps are filled, the answer is just the task count; otherwise the remaining gaps are unavoidable idle intervals.
- **Handles edge cases:** Correctly processes boundary conditions and special inputs
- **Time-space optimization:** Balances time complexity gains with reasonable space usage
